# 课程竞赛大作业：矩阵乘法加速器

## 📁 目录结构

本项目包含三个主要文件夹和一个 Makefile：

- **`code/`**: 包含所有的 Verilog 源代码、Python 测试脚本和 C++ 仿真文件。
  - `testbench_top.v`: 顶层测试台。负责从 `input_mem.csv` 读取输入矩阵并输出到 `result_mem.csv`。实例化了 `ramA`、`ramB`、`ramC`、两个 `buffer` 以及核心 `accelerator`。
  - `accelerator.v`: 封装好的矩阵乘法加速器核心模块（不包含 RAM 部分）。
  - `ram.v`: 存储器模块（基于原项目 `mem.v` 修改），用于实例化 RAM 和 Buffer。
  - `InputGen.py`: 输入矩阵生成脚本。被修改为按 $P \times P$ 瓦片（Tile）进行行主序重排，每行输出 P 个数据（即 $2P$ 个十六进制字符）。
  - `CheckResult.py`: 结果校验脚本。改进了输入等价性检验算法并支持 32-bit（8个十六进制字符）全面精度的输出矩阵校验。
  - `sim_main.cpp`: 用于 Verilator 仿真的主程序。
  - \*.v: 包含 `central_control`、PE 阵列及所需各类基本逻辑组件模块代码。
- **`result/`**: 包含不同脉动阵列尺寸下（$P=2, 4, 8$）加速器的综合原始报告及分数计算草稿。
- **`scripts/`**: 用于逻辑综合的 tcl 脚本。(已将此部分删除)
- **`Makefile`**: 用于驱动逻辑综合操作（基于 Lab 7 修改）。

## 🚀 架构说明与设计思路

本加速器旨在对 $512 \times 512$ 等大型稀疏或密集矩阵进行高效的乘加运算，其核心思路是**分块（Tiling）和脉动阵列（Systolic Array）**。

1. **分块运算**: 大矩阵被拆分为数个 $P \times P$ 的子矩阵（瓦片）。
2. **时序偏斜与运算**: 对于每一小块矩阵运算 $A_{ik} \times B_{kj}$，数据按时序偏斜（Temporal Skew）排列至 `bufferA` 和 `bufferB` 中，然后送入脉动阵列进行流水运算。
3. **累加存储**: 运算结果累加至 `ramC` 中相应部分 $C_{ij}$。重复此过程直至完整矩阵运算完毕。
4. **超参数配置**: 在系统设计中贯穿了一个超参数 **P**（脉动阵列边的尺寸）。通过修改 Testbench 里的这一个超参数，即可快速调整架构面积进行性能与功耗指标测试。
5. **黑盒化控制**: 各个子流程（如时序偏斜、读写、计算）被封装为含 `start` / `done` 握手信号的黑盒模块，使得顶层使用 `central_control` 状态机做总体协调具有极高的复用性和清晰度。

## 🛠️ 仿真与验证流程

建议在运行流程前确保环境中装有 Verilator 或 Icarus Verilog，以及 Python 相关库。系统采用全精度计算，预期最终测试的平方误差和 (SSE) 皆为 $0$。

```bash
# 1. 生成打乱排布的 P×P 瓦片输入矩阵及参考结果
python InputGen.py

# 2. 编译并进行仿真测试（推荐使用 Verilator，若使用 Icarus Verilog 则执行以下命令）
iverilog -o test testbench_top.v
vvp -n test

# 3. 校验输出矩阵结果准确性
python CheckResult.py
```

## 📊 综合成绩与优化分析

经过流片级特性的快速估计及逻辑综合，分别测试了 $P=8$、$P=4$ 和 $P=2$ 配置下的性能（PPA）：
- **综合方式**：使用预设定的逻辑综合流程搭配 PDK。
- **得分表现**：
  - $P=8$: Score $\approx 1.1 \times 10^{14}$
  - $P=4$: Score $\approx 3.846 \times 10^{13}$
  - $P=2$: 未合并 RAM 端口: $\approx 5.761 \times 10^{13}$; 理想状态下合并 RAM: $\approx 1.9 \times 10^{13}$。

**架构权衡思考**：
时间上随 $P$ 的减小线性或多项式地延长，但面积和功率均按 $P^2$ 的速率递减。然而，在现有架构设计中由于 `buffer` 本身的面积主导了加速器尺寸规模，使得当 $P < 2$ 时面积收益出现边际递减甚至由于运行时间的大幅拉长导致总 Score 不降反升。因此设计最优平衡取点基本维持在 $P=2$ 或 $P=4$。
同时，数据搬运/存储器取数操作依旧占据了绝大多数延迟时间，是未来改进的关键性能瓶颈。
